# Model Benchmark

## Introduction

This project benchmarks compiled SiMa Neat model packages with `pyneat.Model.benchmark()`.

The benchmark uses synthetic inputs generated by Neat. It is useful for comparing compiled model
runtime throughput on the DevKit, but it does not measure camera ingest, RTSP decode, image
preprocessing outside the model runner, postprocessing, overlays, encoding, UDP output, or viewer
latency.

## About Project

- Application: `./main.py`
- Runtime API: `pyneat.Model.benchmark()`
- Input: synthetic model input
- Output: console metrics and JSON benchmark reports
- Models: `./assets/models/*.tar.gz` (git-ignored)
- Reports: `./output/*_benchmark.json`

Current asset models:

```text
./assets/models/yolo_v8n_mpk.tar.gz
./assets/models/yolo_v8m_mpk.tar.gz
./assets/models/yolo_v8n_seg_mpk.tar.gz
```

## Requirements

Run from an environment where `dk` is configured for the Modalix DevKit.


The script runs on the DevKit through `dk`, where `pyneat` is available. No CMake build is required
because this is a Python benchmark utility.

Check runner availability:

```bash
command -v dk
```


Run the commands below from this app folder:

```bash
cd /path/to/demo-neat/apps/benchmark
```

## Model Download Commands

These benchmark commands need compiled model packages in `./assets`. Download the Model Zoo
packages once from the SDK shell:

```bash
cd /path/to/demo-neat/apps/benchmark
mkdir -p ./assets
cd ./assets
sima-cli modelzoo -v 2.1.2 --boardtype modalix get yolo_v8n
sima-cli modelzoo -v 2.1.2 --boardtype modalix get yolo_v8m
sima-cli modelzoo -v 2.1.2 --boardtype modalix get yolo_v8n_seg
cd ..
```

If the shell prints `getcwd: cannot access parent directories`, the terminal is still in a folder
that was moved or deleted. Run `cd /path/to/demo-neat/apps/benchmark` again and retry the download.

Expected model files:

```text
./assets/models/yolo_v8n_mpk.tar.gz
./assets/models/yolo_v8m_mpk.tar.gz
./assets/models/yolo_v8n_seg_mpk.tar.gz
```

If your DevKit uses a different platform version, replace `2.1.2` with that release version.

## Model Assets

Expected model directory (git-ignored — you download or build the archives):

```text
./assets/models/
```

The benchmark command accepts any compiled model package:

```bash
--model ./assets/models/<model>.tar.gz
```

## Configure

`./config/default.yaml` supplies the defaults when you omit the CLI flags:

```yaml
model:
  path: ./assets/models/yolo_v8n_mpk.tar.gz
benchmark:
  frames: 1000
output:
  report_json: ./output/yolo_v8n_benchmark.json
```

Every value is overridable on the command line (`--model`, `--frames`, `--output-json`), and the
"How To Run" commands below do exactly that.

Reports are written to `./output/` (git-ignored).

## How To Run

Run YOLOv8n:

```bash
dk ./main.py \
  --model ./assets/models/yolo_v8n_mpk.tar.gz \
  --frames 1000 \
  --output-json ./output/yolo_v8n_benchmark.json
```

Run YOLOv8m:

```bash
dk ./main.py \
  --model ./assets/models/yolo_v8m_mpk.tar.gz \
  --frames 1000 \
  --output-json ./output/yolo_v8m_benchmark.json
```

Run YOLOv8n segmentation:

```bash
dk ./main.py \
  --model ./assets/models/yolo_v8n_seg_mpk.tar.gz \
  --frames 1000 \
  --output-json ./output/yolo_v8n_seg_benchmark.json
```

## How To Read The Output

Console output includes:

```text
latency_ms=<average model latency>
fps=<synthetic model throughput>
avg_power_watts=<reported benchmark power, if available>
energy_joules=<reported benchmark energy, if available>
report_json=<written report path>
```

Each JSON report stores:

```text
benchmark.frames
benchmark.timestamp_utc
model.path
model.file
model.input_specs
model.output_specs
metrics.latency_ms
metrics.fps
metrics.avg_power_watts
metrics.energy_joules
```

## Notes And Caveats

- This is not an end-to-end application benchmark.
- It does not include RTSP/camera frame acquisition.
- It does not include decoded detection parsing, NMS, segmentation mask projection, overlays, H.264
  encoding, UDP transport, or display.
- Use this benchmark to compare compiled model packages under the same synthetic Neat benchmark
  conditions.
- Use the single-stream demo applications when you need full pipeline FPS or latency.
